3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
Шаблони (патерни) проєктування – це формалізовані, багаторазово перевірені архітектурні рішення типових проблем, що виникають у процесі розробки програмного забезпечення. Патерн не є фрагментом коду, який можна безпосередньо інтегрувати в програму. Він є концептуальною моделлю, що описує оптимальний спосіб взаємодії класів та об’єктів. Відповідно до класифікації авторів книги «Design Patterns» («Банда чотирьох», GoF), шаблони поділяються на три основні категорії:
* породжувальні (creational): абстрагують процес ініціалізації об’єктів, приховуючи логіку їх створення;
* структурні (structural): визначають правила побудови гнучких ієрархій та ефективних зв'язків між класами;
* поведінкові (behavioral): регламентують алгоритми взаємодії та розподіл обов’язків між об’єктами системи.
3.2 Опис обраного патерна
Назва патерна: Facade (Фасад)
Тип шаблона (патерна): структурний
Основне призначення: надання уніфікованого, спрощеного інтерфейсу високого рівня до набору різноманітних інтерфейсів у складній підсистемі. Патерн приховує внутрішню складність реалізації, роблячи підсистему доступнішою для клієнтського коду.
Передумови застосування:
* наявність складної сторонньої бібліотеки або застарілої (legacy) системи, з якої клієнту потрібна лише базова функціональність;
* необхідність зниження рівня залежностей (декаплінг) між клієнтським кодом і компонентами підсистеми;
* проєктування багаторівневої (layered) архітектури, де комунікація між рівнями має відбуватися виключно через єдину точку входу.
Переваги:
* надійна ізоляція клієнтів від складних складових підсистеми;
* забезпечення слабкого зв’язування (weak coupling), що спрощує незалежний розвиток модулів;
* покращення можливостей для модульного тестування (unit testing) шляхом створення мок-об’єктів для інтерфейсу Фасаду.
Недоліки:
* існує ризик перетворення Фасаду на антипатерн «Божественний об’єкт» (God Object), якщо сконцентрувати в ньому занадто багато відповідальності та функцій.
3.3 Структура шаблона (патерна)
Опис структурних частин:
* facade (фасад): клас, що надає спрощений доступ до функціональності підсистеми. Відповідає за переадресацію запитів клієнта до відповідних об’єктів підсистеми;
* subsystem classes (класи підсистеми): комплекс класів, що виконують безпосередню роботу (наприклад, кодеки, парсери). Вони функціонують незалежно і не містять посилань на клас Фасаду;
* client (клієнт): програмний компонент, який використовує Фасад замість прямої взаємодії з множиною об’єктів підсистеми.
Опис взаємодії об’єктів:
Клієнт ініціює взаємодію, викликаючи відповідний метод Фасаду. Фасад аналізує запит, створює необхідні екземпляри класів підсистеми, налаштовує їх і делегує їм виконання задачі. Класи підсистеми виконують операцію, після чого Фасад (за потреби) збирає результати та повертає їх Клієнту.
UML-діаграма з поясненнями:


  

Рисунок 3.1 – UML-діаграма патерна Facade        


На діаграмі ліворуч продемонстровано проблему високої зв’язності, коли клієнти звертаються до багатьох класів підсистеми напряму. Праворуч зображено рішення із застосуванням патерна: впроваджується проміжний клас Facade. Клієнти взаємодіють виключно з ним. Стрілки залежностей вказують від Фасаду до підсистеми, підтверджуючи незалежність останньої.
3.4 Реалізація шаблона (патерна)
Опис реалізації мовою програмування Java.
Основні етапи:
1. створення базових інтерфейсів та класів: розроблено імітацію складної підсистеми конвертації відео, що включає класи VideoFile, Codec, BitrateReader та AudioMixer;
2. реалізація конкретних класів (Фасаду): створено клас VideoConverter, який інкапсулює алгоритм роботи з підсистемою у межах одного методу convertVideo;
3. демонстрація використання: написано клієнтський клас MainApp, який виконує конвертацію відео, звертаючись виключно до Фасаду, не ініціалізуючи внутрішні класи підсистеми.

5 this.name = name;
6 }
7 }

9 interface Codec { }
10
11 class MPEG4CompressionCodec implements Codec { }
12
13 class OggCompressionCodec implements Codec { }
14
15 class CodecFactory {
16 public static Codec extract(VideoFile file) {
17 return new MPEG4CompressionCodec();
18 }
19 }
20
21 class BitrateReader {
22 public static VideoFile read(VideoFile file, Codec codec) {
23 return file;
24 }
25 public static VideoFile convert(VideoFile buffer, Codec codec) {
26 return buffer;
27 }
28 }
29
30 class AudioMixer {
31 public Object fix(VideoFile result) {
32 return new Object();
33 }
34 }
35
36 // Реалізація Фасаду
37 class VideoConverter {
38 public Object convertVideo(String fileName, String format) {
39 VideoFile file = new VideoFile(fileName);
40 Codec sourceCodec = CodecFactory.extract(file);
41 Codec destinationCodec;
42
43 if (format.equals("mp4")) {
44 destinationCodec = new MPEG4CompressionCodec();
45 } else {
46 destinationCodec = new OggCompressionCodec();
47 }
48
49 VideoFile buffer = BitrateReader.read(file, sourceCodec);
50 VideoFile intermediateResult = BitrateReader.convert(buffer, destinationCodec);
51 return (new AudioMixer()).fix(intermediateResult);
52 }
53 }
54
55 // Демонстрація використання (Клієнт)
56 public class MainApp {
57   public static void main(String[] args) {
58     // Клієнт взаємодіє лише з Фасадом, складність прихована
59     VideoConverter converter = new VideoConverter();
60     
61     // Вся складність схована за одним викликом
62     Object mp4Video = converter.convertVideo("youtubevideo.ogg", "mp4");
63     
64     System.out.println("Відео успішно конвертовано!");
65   }
